## 1. Why a TypeScript client?

Daml smart contracts live on the **ledger** (Canton's shared record). On their own, contracts do nothing: they wait for an external system to send commands — create a contract, run a choice, query the active state.

Canton provides an **HTTP API** so any external system can do exactly that. TypeScript is a natural choice for building that client because:

- It is the most common language for modern financial application backends.
- It has a strong type system that helps us match Daml types to TypeScript structures at compile time.
- Node.js has built-in `fetch` since version 18, which makes HTTP calls simple.

The separation is on purpose: **Daml contracts define the rules**, and **the TypeScript client decides who runs what and when**. Neither can exist alone.

## 3. How TypeScript talks to Canton

### The JSON Ledger API v2

Canton provides an HTTP API called the **JSON Ledger API**. In SDK 3.x (the version used here), the base path is `/v2/` — the old `/v1/` path was removed.

> **Important:** This caused problems early in the project. The sandbox starts on port `7575` by default, but the `/v1/*` endpoints return `404`. You must use `/v2/*`.

The endpoints we use:

| Operation | Method | Endpoint |
|---|---|---|
| Create a contract | `POST` | `/v2/commands/submit-and-wait-for-transaction` |
| Run a choice | `POST` | `/v2/commands/submit-and-wait-for-transaction` |
| Query active contracts | `GET` + `POST` | `/v2/state/ledger-end` + `/v2/state/active-contracts` |
| List parties | `GET` | `/v2/parties` |
| Register a party | `POST` | `/v2/parties` |
| Health check | `GET` | `/docs/openapi` |

### The command format: proto-transcoding

The Canton v2 API is not a standard REST API. Internally it uses gRPC + Protocol Buffers, and the HTTP API is a translation of that format. This has a direct effect on the JSON body structure:

The command fields **do not go at the root** of the body. They go inside an object called `commands` (which maps to the `Commands commands` field in the proto message `SubmitAndWaitRequest`):

```json
// ❌ Wrong format (the server returns HTTP 400):
{
  "commandId": "cmd-123",
  "actAs": ["Alice::1220..."],
  "commands": [{ "CreateCommand": { ... } }]
}

// ✅ Correct format (proto-transcoding):
{
  "commands": {
    "commandId": "cmd-123",
    "userId": "my-app",
    "actAs": ["Alice::1220..."],
    "readAs": ["Alice::1220..."],
    "applicationId": "my-app",
    "commands": [{ "CreateCommand": { ... } }]
  }
}
```

The outer `commands` field is the `SubmitAndWaitRequest` message. The inner `commands` (array) is the list of individual commands.

## 4. Types: TypeScript mirrors Daml

The file `src/types/contracts.ts` has two jobs:

1. **TypeScript interfaces** that copy the fields of each Daml template.
2. **Envelope types** that model what the API returns (contracts, events, results).

### Contract interfaces

Each TypeScript interface maps directly to a Daml template:

```typescript
// In Daml (Asset.daml):
// template Asset
//   with
//     issuer   : Party
//     owner    : Party
//     symbol   : Text
//     quantity : Decimal
//     observers: [Party]

// In TypeScript (contracts.ts):
export interface Asset {
  issuer:    string;
  owner:     string;
  symbol:    string;
  quantity:  DamlDecimal;  // string, not number
  observers: string[];
}
```

> **Why is `quantity` a `string` and not a `number`?**
>
> Daml represents `Decimal` values as JSON strings to avoid floating-point rounding errors. A JavaScript `number` cannot exactly represent values like `0.1` + `0.2`. In financial systems, this matters a lot. That is why we have the type `DamlDecimal = string` and the helper function `toDecimalString(n)`.

### Template IDs

Every Daml contract lives under a unique template identifier. In the v2 API, the format is:

```
#<package-name>:<Module>:<Template>
```

The `#` means "look up by package name" (instead of by package hash). This is more stable when the contract is updated:

```typescript
export const TEMPLATE_IDS = {
  ASSET:            '#asset-swap-contracts:Asset:Asset',
  SWAP_PROPOSAL:    '#asset-swap-contracts:SwapProposal:SwapProposal',
  SWAP_SETTLEMENT:  '#asset-swap-contracts:SwapProposal:SwapSettlement',
  TRANSFER_REQUEST: '#asset-swap-contracts:TransferBatch:TransferRequest',
  TRANSFER_BATCH:   '#asset-swap-contracts:TransferBatch:TransferBatch',
} as const;
```

`asset-swap-contracts` is the `name` in the `daml.yaml` of the contracts package.

### Envelope types

The API returns contracts and events wrapped in response structures. These types model them:

```typescript
// An active contract on the ledger
export interface Contract<T> {
  contractId: string;   // the unique ID of the contract (e.g. "00ff13...")
  payload:    T;        // the contract fields (typed by the template)
  templateId: string;   // the template ID (e.g. "#asset-swap-contracts:Asset:Asset")
}

// The result of running a choice
export interface ExerciseResult<R> {
  exerciseResult: R;          // the return value of the choice
  events:         LedgerEvent[]; // all events generated (created / archived)
}

// A ledger event (union type)
export type LedgerEvent =
  | { created:  Contract<unknown> }
  | { archived: { contractId: string; templateId: string } };
```

## 5. The `LedgerClient`: the core of everything

`src/ledger/client.ts` has the `LedgerClient` class. It is a thin wrapper around Canton's HTTP API: it has no business logic, it only knows how to talk to the ledger.

### One instance per party

Each `LedgerClient` instance is **tied to one single party**:

```typescript
const aliceClient = new LedgerClient(
  'http://localhost:7575',  // API URL
  aliceToken,              // Alice's JWT token
  alicePartyId,            // "Alice::1220abc..."
  'my-app',                // userId (applicationId)
);
```

This reflects how security works in Daml: every command says explicitly **on whose behalf** it runs (`actAs`). If Alice tries to run a choice controlled by Bob, Daml rejects the transaction.

### `create<T>` — Create a contract

```typescript
const asset = await client.create<Asset>(
  TEMPLATE_IDS.ASSET,
  { issuer: aliceId, owner: aliceId, symbol: 'TokenX', quantity: '200', observers: [] }
);
// asset.contractId → "00ff137148ce61..."
// asset.payload    → { issuer: "Alice::...", owner: "Alice::...", ... }
```

Internally, it does `POST /v2/commands/submit-and-wait-for-transaction` and pulls the `CreatedEvent` from the response:

```typescript
async create<T>(templateId: string, createArguments: T): Promise<Contract<T>> {
  const result = await this.post('/v2/commands/submit-and-wait-for-transaction', {
    commands: {
      commandId:     this.nextCommandId(),
      userId:        this.userId,
      actAs:         [this.partyId],
      readAs:        [this.partyId],
      applicationId: this.userId,
      commands: [{ CreateCommand: { templateId, createArguments } }],
    },
  });

  // The response has the transaction with the events it generated
  const createdEvent = result.transaction.events
    .filter(e => 'CreatedEvent' in e)[0].CreatedEvent;

  return {
    contractId: createdEvent.contractId,
    payload:    createdEvent.createArgument as T,
    templateId: createdEvent.templateId,
  };
}
```

### `exercise<A, R>` — Run a choice

```typescript
const result = await client.exercise<
  { counterpartyAssetCid: string },  // A = type of the choice argument
  string                             // R = return type of the choice
>(
  TEMPLATE_IDS.SWAP_PROPOSAL,
  proposalContractId,
  'Accept',
  { counterpartyAssetCid: tokenYId },
);
// result.exerciseResult → "00bb8aae..."  (the new SwapSettlement)
// result.events         → [ArchivedEvent(proposal), CreatedEvent(settlement)]
```

#### The tree endpoint problem and the solution

The v2 API has two endpoints for commands:
- `/v2/commands/submit-and-wait-for-transaction` → returns a flat transaction with events.
- `/v2/commands/submit-and-wait-for-transaction-tree` → returns a tree of events with the choice return value built in.

The tree endpoint has a different body format in `dpm sandbox 3.4.x`. To avoid that problem, `exercise()` uses the flat endpoint and **gets the `exerciseResult` from the `CreatedEvent`s generated**:

| New contracts created | `exerciseResult` | Covers |
|---|---|---|
| 0 | `null` | Choices returning `()` — Cancel, Abort |
| 1 | `string` (the contractId) | `ContractId T` — Accept, Transfer, Merge |
| N | `string[]` | `(ContractId T, ContractId T)` or `[ContractId T]` — Settle, ExecuteTransfers |

This works for all choices in the project because **all of them return contract IDs**, and the ledger generates events in the same order that Daml creates them.

### `query<T>` — Query active contracts

Querying the active state of the ledger needs two steps:

1. **Get the current offset** (`GET /v2/state/ledger-end`) — the ledger's current position.
2. **Get the contracts** (`POST /v2/state/active-contracts`) — contracts alive _up to that offset_.

```typescript
// Step 1: get the offset
const { offset } = await fetch('/v2/state/ledger-end');

// Step 2: filter by template and party
const contracts = await this.post('/v2/state/active-contracts', {
  filter: {
    filtersByParty: {
      [this.partyId]: {
        cumulative: [{
          identifierFilter: {
            TemplateFilter: {
              value: { templateId, includeCreatedEventBlob: false }
            }
          }
        }]
      }
    }
  },
  verbose: true,
  activeAtOffset: offset,
});
```

After that, a field filter is applied on the client side to keep only the contracts that match the given fields:

```typescript
const aliceAssets = await client.query<Asset>(
  TEMPLATE_IDS.ASSET,
  { owner: alicePartyId }  // filter applied in TypeScript (client-side)
);
```

> **Note:** The field filter (e.g. `{ owner: alicePartyId }`) runs in TypeScript after receiving all contracts of that template. For very large sets of contracts, a server-side filter would be faster, but for the local sandbox this is fine.

## 6. The Role pattern: each party has its own class

Instead of exposing the `LedgerClient` directly, the project defines **role classes** that group _what each type of participant can do_. This has three benefits:

1. **Clarity**: reading `alice.proposeSwap(...)` is clearer than reading `aliceClient.create(TEMPLATE_IDS.SWAP_PROPOSAL, { ... })`.
2. **Safety**: if you try to call `settler.proposeSwap()` by mistake, TypeScript tells you that method does not exist.
3. **Easier to change**: if a Daml template changes, you only need to update the matching role class.

### `AssetOwner` — The asset owner

Represents what Alice or Bob can do as holders of their own tokens:

```typescript
class AssetOwner {
  async createAsset(params)           // Issue (mint) a new asset
  async splitAsset(contractId, qty)   // Split into two (UTXO pattern)
  async mergeAssets(primary, other)   // Combine two into one (UTXO pattern)
  async discloseAsset(cid, observer)  // Give visibility to another party
  async proposeSwap(params)           // Propose an exchange
  async cancelProposal(cid)           // Cancel a pending proposal
  async authorizeTransfer(params)     // Give the operator permission for a transfer
  async cancelTransferRequest(cid)    // Cancel a transfer permission
  async queryAssets()                 // See all my active assets
}
```

### `Counterparty` — Whoever responds to a proposal

Represents the view of whoever receives a swap offer (Bob when he responds to Alice):

```typescript
class Counterparty {
  async acceptProposal(proposalCid, counterpartyAssetCid)  // Accept → creates SwapSettlement
  async rejectProposal(proposalCid)                        // Reject → archives the proposal
  async queryProposals()                                   // See proposals sent to me
}
```

### `Settler` — The Operator

Has two different jobs:

```typescript
class Settler {
  // Atomic settlement:
  async settleSwap(settlementCid)         // Run the swap in 1 transaction
  async abortSwap(settlementCid)          // Abort the settlement

  // Batching:
  async createTransferBatch(requestCids)  // Group N TransferRequests
  async executeTransferBatch(batchCid)    // Run them all in 1 roundtrip
  async cancelTransferBatch(batchCid)     // Cancel the batch without running it

  // Queries:
  async queryPendingSettlements()         // SwapSettlements waiting for me
  async queryPendingTransferRequests()    // TransferRequests where I am the operator
}
```

### How the roles are created

Each role gets a `LedgerClient` that is already set up with the right token and party ID:

```typescript
// Each party has their own client (and therefore their own JWT token)
const aliceClient    = new LedgerClient(baseUrl, aliceToken, alicePartyId);
const bobClient      = new LedgerClient(baseUrl, bobToken, bobPartyId);
const operatorClient = new LedgerClient(baseUrl, operatorToken, operatorPartyId);

// The roles wrap the matching client
const alice             = new AssetOwner(aliceClient, alicePartyId);
const bob               = new AssetOwner(bobClient, bobPartyId);
const bobAsCounterparty = new Counterparty(bobClient, bobPartyId);
const operator          = new Settler(operatorClient, operatorPartyId);
```

Bob is both an `AssetOwner` (he can issue and manage his tokens) and a `Counterparty` (he can respond to proposals). The same Bob `LedgerClient` is reused in both roles — no need to duplicate credentials.

## 7. Authentication: JWT tokens in sandbox mode

### What is a JWT?

A JWT (_JSON Web Token_) is a signed token that contains _claims_ (pieces of information). In Canton, the JWT tells the ledger **who you are** (your identity/userId).

In production, JWTs are issued by an external identity system (OAuth2, etc.). In the development sandbox, Canton accepts any well-formed JWT because `dpm sandbox` runs with relaxed authentication.

### The token format for Canton v2

```typescript
function generateDevToken(partyId: string): string {
  const displayName = partyId.split('::')[0]; // "Alice" from "Alice::1220abc..."

  const payload = {
    sub:   displayName,       // user ID — the server uses this to identify the user
    scope: 'daml_ledger_api', // tells the server this token is for the Ledger API
    aud:   [],                // audiences (empty in sandbox)
    iss:   null,              // issuer (null in sandbox)
  };

  return jwt.sign(payload, DEV_JWT_SECRET, { algorithm: 'HS256' });
}
```

The `sub` (subject) field is the `userId`. In sandbox mode, the server accepts any valid `sub`. The important thing is that it matches the `userId` we send in the commands.

### Why userId also goes in the body

Even though the JWT already has the `sub`, the v2 API also requires `userId` to be set explicitly in the command body. This lets the server check that the token and the command are consistent:

```json
{
  "commands": {
    "userId":  "Alice",      // must match the sub in the JWT
    "actAs":   ["Alice::1220..."],
    "readAs":  ["Alice::1220..."],
    ...
  }
}
```

> **Lesson learned:** Leaving out `userId` from the body gives an error `HTTP 400: The submitted request is missing a user-id`. Even if the JWT has a valid `sub`, the v2 API needs the field written out explicitly.

### How the token is sent

The token is included in every request as a standard HTTP header:

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

In the `LedgerClient`:

```typescript
private get authHeaders(): Record<string, string> {
  return {
    'Content-Type':  'application/json',
    'Authorization': `Bearer ${this.token}`,
  };
}
```

## 8. The local environment: `setup.sh` and `setup-env.ts`

Before running the demos, you need to:
1. Start the sandbox (the local ledger).
2. Register the parties (Alice, Bob, Operator).
3. Generate the JWT tokens and save them to the `.env` file.

This is split into two scripts that run in order.

### `scripts/setup.sh` — Start the sandbox

The shell script does four things:

```bash
# 1. Compile the Daml contracts to .dar
dpm build

# 2. Start the sandbox with the HTTP API on port 7575
dpm sandbox \
  --json-api-port 7575 \
  --ledger-api-port 6865 \
  --dar asset-swap-contracts-0.1.0.dar &

# 3. Wait for the API to be ready (health check at /docs/openapi)
until curl -s /docs/openapi > /dev/null; do sleep 2; done

# 4. Register the parties via HTTP
curl -X POST /v2/parties -d '{"partyIdHint": "Alice", "identityProviderId": ""}'
curl -X POST /v2/parties -d '{"partyIdHint": "Bob",   "identityProviderId": ""}'
curl -X POST /v2/parties -d '{"partyIdHint": "Operator", "identityProviderId": ""}'
```

#### Why `dpm` and not `daml`?

In SDK 3.4+, the `daml` CLI was deprecated. The replacement is `dpm` (_Digital Asset Package Manager_). `daml start` showed deprecation warnings and had different behavior (like stopping the sandbox automatically after 2 minutes). `dpm sandbox` is the official and stable command.

#### Why is the health check at `/docs/openapi`?

The `/docs/openapi` endpoint returns the API's OpenAPI specification without needing authentication. Other endpoints like `/v2/parties` or `/v2/state/ledger-end` need a JWT token. Using `/docs/openapi` keeps the health check simple.

#### Why the `.dar` file and not the multi-package?

`dpm sandbox` can start the ledger with just the `.dar` from the business contracts package (`asset-swap-contracts`), without needing to compile or run the scripts package (`asset-swap-scripts`). This is cleaner: the Daml scripts are only for unit tests, not for the integration sandbox.

### `src/scripts/setup-env.ts` — Generate the `.env`

This TypeScript script does three things:

**Step 1: Find or create the parties**

```typescript
// Try to get Alice, Bob, and Operator from the ledger
const response = await fetch('/v2/parties');
const parties  = parsePartyMap(response);

// If they do not exist, create them automatically
if (!parties.has('Alice')) {
  const partyId = await allocateParty('Alice');
  // POST /v2/parties { partyIdHint: "Alice" }
}
```

Party IDs in Canton follow the format `DisplayName::hash` — for example, `Alice::1220f1ad80b...`. The hash is unique per sandbox and changes on every restart. That is why they cannot be hardcoded.

**Step 2: Generate JWT tokens**

For each party, it generates an HS256 token with `sub` equal to the display name:

```typescript
const token = jwt.sign(
  { sub: 'Alice', scope: 'daml_ledger_api', aud: [], iss: null },
  'canton-dev-insecure-secret-do-not-use-in-production',
  { algorithm: 'HS256' }
);
```

**Step 3: Write the `.env`**

```
LEDGER_JSON_API_URL=http://localhost:7575

ALICE_PARTY=Alice::1220f1ad80bda06e78b0c19668831445245d0424ae5b9756f7b229ec6f0bdf3d1674
ALICE_TOKEN=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

BOB_PARTY=Bob::1220f1ad80bda06e78b0c19668831445245d0424ae5b9756f7b229ec6f0bdf3d1674
BOB_TOKEN=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

OPERATOR_PARTY=Operator::1220f1ad80bda06e78b0c19668831445245d0424ae5b9756f7b229ec6f0bdf3d1674
OPERATOR_TOKEN=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

### `src/config.ts` — Load the configuration

`loadConfig()` reads the `.env` and checks that all required variables are present. If any are missing, it throws a clear error before the demo fails with a confusing message:

```typescript
export function loadConfig() {
  return {

        token: required('ALICE_TOKEN'),
      },
      // ...
    },
  } as const;
}
```

## 9. The demos: the full flow in action

### `demo-swap.ts` — Asset Swap in 4 steps

```
Step 1 — Issue assets
Step 2 — Alice proposes a swap
Step 3 — Bob accepts the proposal
Step 4 — Operator settles (atomic 2-leg transfer)
Verification — Final balances
```

Let us look at each step with the real code:

#### Step 1: Issue assets

```typescript
const tokenX = await alice.createAsset({
  symbol:    'TokenX',
  quantity:  200,
  observers: [bobPartyId, operatorPartyId],  // they need to be able to see the contract
});

const tokenY = await bob.createAsset({
  symbol:    'TokenY',
  quantity:  100,
  observers: [alicePartyId, operatorPartyId],
});
```

The observers matter: when Alice proposes the swap, Bob and the Operator need to be able to _see_ the `Asset` that Alice is offering.

#### Step 2: Alice proposes

```typescript
const proposal = await alice.proposeSwap({
  counterparty:      bobPartyId,
  settler:           operatorPartyId,
  offeredAssetCid:   tokenX.contractId,  // the ContractId of TokenX
  offeredSymbol:     'TokenX',
  offeredQuantity:   200,
  requestedSymbol:   'TokenY',
  requestedQuantity: 100,
});
```

This creates a `SwapProposal` contract on the ledger. The contract stores a reference to `TokenX` through its `contractId` — it does not copy the content.

#### Step 3: Bob accepts

```typescript
const acceptResult = await bobAsCounterparty.acceptProposal(
  proposal.contractId,
  tokenY.contractId,   // Bob offers his TokenY in return
);

const settlementContractId = acceptResult.exerciseResult;
```

`exerciseResult` holds the ID of the new `SwapSettlement` that was created. This contract is what the Operator needs to settle the trade.

#### Step 4: The Operator settles

```typescript
const settleResult = await operator.settleSwap(settlementContractId);

const [newTokenXId, newTokenYId] = settleResult.exerciseResult;
```

One HTTP call, one Daml transaction: Alice's `TokenX` goes to Bob and Bob's `TokenY` goes to Alice. If anything fails in the middle, **both** transfers are rolled back.

#### Verification

```typescript
const [aliceAssets, bobAssets] = await Promise.all([
  alice.queryAssets(),
  bob.queryAssets(),
]);

// Alice: 200.0 TokenY  (received Bob's TokenY)
// Bob:   100.0 TokenX  (received Alice's TokenX)
```

### `demo-batch.ts` — Batch Transfer

This demo shows the batching pattern: transfer 3 assets from Alice to Bob in **1 ledger roundtrip** instead of 3:

```
Step 1 — Alice issues 3 assets (TokenA, TokenB, TokenC)
Step 2 — Alice creates 3 TransferRequests (gives the Operator permission)
Step 3 — Operator creates a TransferBatch (groups the 3 requests)
Step 4 — Operator runs the batch (1 transaction, 3 transfers)
```

#### Step 2: Transfer permissions

```typescript
const reqA = await alice.authorizeTransfer({
  operator: operatorPartyId,   // the Operator can run this
  newOwner: bobPartyId,        // going to Bob
  assetCid: tokenA.contractId,
});
```

Each `TransferRequest` is a contract created by the asset owner (Alice) that says: _"Operator: you are allowed to transfer this asset to Bob."_ The Operator is just an observer at this point — they have not done anything yet.

#### Step 3: Create the batch

```typescript
const batch = await operator.createTransferBatch([
  reqA.contractId,
  reqB.contractId,
  reqC.contractId,
]);
```

The `TransferBatch` is another contract that groups the references to the `TransferRequest`s. It still does not run anything.

#### Step 4: Run it

```typescript
const batchResult = await operator.executeTransferBatch(batch.contractId);
const newAssetIds = batchResult.exerciseResult; // [idA, idB, idC]
```

**One HTTP call, one Daml transaction, 3 transfers**. The `ExecuteTransfers` choice goes through the list of requests, calls `Transfer` on each asset, and returns the list of new contract IDs. If any transfer fails (for example, the asset was already consumed), the whole transaction is rolled back.

## 11. Design decisions and lessons learned

### Why is the project split into two folders?

```
canton-temporal-ai/
├── daml-contracts/   ← Daml contracts (compiled with dpm)
└── ts-client/        ← TypeScript client (run with Node.js)
```

They could live in the same repo, but are kept in separate folders for one main reason: **they use completely different languages, tools, and runtimes**.

- `daml-contracts` uses the Daml compiler, `dpm build`, and produces `.dar` files.
- `ts-client` uses Node.js, `pnpm`, TypeScript, and connects to the ledger over HTTP.

If they were mixed together, we would have build tool conflicts and unclear responsibilities. The separation also reflects how real teams work: the smart contract engineers and the backend engineers are rarely the same people.

### Why `pnpm` and not `npm`?

`pnpm` is a more efficient package manager than `npm`: it uses a central store with hard links instead of copying `node_modules` into each project. On a machine with multiple projects, the disk space and installation time savings are meaningful.

### Why not hardcode the credentials?

Canton party IDs contain a hash that is generated when the party is registered on the ledger. That hash **changes every time the sandbox restarts** because it is a fresh ledger instance. That is why they cannot be hardcoded in the code or in version-controlled configuration files.

The solution: `setup-env.ts` reads the party IDs from the running ledger and generates the `.env` file fresh for each session.

### The tree endpoint workaround

The `/v2/commands/submit-and-wait-for-transaction-tree` endpoint exists to return the `exerciseResult` (the return value of the Daml choice). In theory it is the right endpoint for `exercise()`. In practice, in `dpm sandbox 3.4.x`, that endpoint uses a different body format that caused `HTTP 400` errors.

The practical fix: use the flat endpoint for everything and get the `exerciseResult` from the `CreatedEvent`s. This works because all choices in the project return `ContractId`s, and the ledger generates events in the same order that Daml runs them.

### Event order is guaranteed

In Daml, when a choice runs multiple sub-choices in sequence, the events in the transaction appear in that same order. This is a guarantee from the Daml runtime and lets us rebuild the return value of choices that return tuples:

```daml
-- Daml: returns (first, second)
choice Settle : (ContractId Asset, ContractId Asset)
  do
    first  <- exercise offeredAssetCid Transfer with newOwner = counterparty
    second <- exercise counterpartyAssetCid Transfer with newOwner = proposer
    return (first, second)
```

```typescript
// TypeScript: the events array follows that same order
const [firstCid, secondCid] = settleResult.exerciseResult;
```

### Conclusion

The pattern that comes out of this project is clear:

```
Daml defines WHAT and WHO.        (the contracts and their permissions)
TypeScript defines WHEN and HOW.  (the flow and the orchestration)
Canton enforces COMPLIANCE.       (rejects what Daml does not allow)
```

Each layer has its place. The Daml contracts know nothing about HTTP. The TypeScript client does not copy the business rules — it just says "run this choice with these arguments". Canton handles the rest.
